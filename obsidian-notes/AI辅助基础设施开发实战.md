## 第一次尝试：失败的147次

**目标**：在 Windows 和 macOS 上编译 Milvus

**问题**：
- Go + C++ + Rust 混合项目
- 陈年 Conan + CMake 构建系统
- 跨平台编译是组里"谁碰谁死"的禁区

**过程**：
- Windows 编译过了 → Linux 挂了 → 修 Linux → Mac 挂了 → 修 Mac → GPU 挂了
- 1000个文件、近10万行代码的巨型补丁
- 147条手动批准的 permission 规则
- 三个 $200 Claude Max 订阅 + 一整个假期

**根本原因**：
> **我没有全局视角，Claude 当然也不会有。**

AI 精确解决我放在它面前的问题，仅此而已。"Windows 编译通过"是局部最优，但我没告诉它真正的目标是"所有平台零 hack 通过"。

## 正确的方法论

### 1. Plan 先行
写任何代码之前，先把整件事想清楚：
- 构建系统分几层？
- 依赖关系什么样？
- 从哪里开始，按什么顺序往上搭？

### 2. 定义约束（最关键）
约束 = "什么叫做完了"
- 测试是约束的一部分
- 不引入 `#ifdef` hack 是约束
- 不破坏 Conan 依赖链是约束
- 不增加不必要的抽象层是约束

> **不是让 AI 写代码，是先定义"什么叫写对了"，再让 AI 去实现。**

### 3. 从底层往上搭，一层一层来
- 别一口气让 AI 改 1000 个文件
- 每层小、每层可验证
- 约束满足了再往上一层

### 4. Review 测试，不 review 代码
- 写测试（或让 AI 生成）
- 审查的是测试，不是实现
- 覆盖所有平台了吗？
- 构建产物对吗？
- 跨模块接口契约还在吗？

**结果对比**：
| 方式 | 时间 | 结果 |
|------|------|------|
| 第一次（错误方法） | 3个订阅+整个假期 | 1000个文件的烂摊子 |
| 第二次（正确方法） | 2天 | 数十个干净提交 + 大量跨平台测试用例 |

## 并行开发：突破吞吐量瓶颈

**问题**：单会话处理 200万行代码库，每个任务 20-30 分钟，干等

**解决方案**：
- 3台机器（服务器 + Mac Mini + MacBook Pro）
- 6个终端，每个独立 Claude Code 会话
- `git worktree` 隔离工作目录
- 每个分支独立任务、独立约束、独立测试

**并行场景**：
- 会话1：解决 Linux ARM Conan 依赖
- 会话2：修 macOS CMake 配置
- 会话3：处理 Windows MSVC 兼容性

> **用 AI 搞基础设施开发，瓶颈从来不是智力，是并行度。**

## 多模型代码审查：Magpie

**问题**：约束和测试保证"能不能用"，保证不了"该不该这么写"

**解决方案**：多个 AI 模型同时审查，互相辩论

**测评结果**（15个真实 PR，全是合入后出问题的）：

| 模式 | 检出率 | L3 级系统 bug |
|------|--------|---------------|
| Claude 裸测 | 53% | 全中 |
| Gemini 裸测 | 13% | - |
| Gemini + 上下文 | 33% | - |
| Claude + 上下文 | 47% | - |
| **四模型辩论（5轮）** | **80%** | **100%** |

**最优组合**：Claude + Gemini（覆盖四模型上限的 91%）

**仍有遗漏**：
15个 bug 里 4 个全军覆没——代码语法完全正确，bug 藏在开发者脑子里的假设中，不在 diff 里，甚至不在周边代码里。

> **能看到这 4 个的，目前还只有人。**

## 真正的收获

### 1. AI 做局部最优，你的工作是做全局最优

AI 精确解决你放在它面前的问题。"Windows 编译通过"是局部最优，但"所有平台零 hack 通过"才是正确的全局目标。

> **构建一个在每个用户、每个平台、每个环境下都能工作的产品，是本质上不同的问题。这个差距，正是工程价值的栖身之所。**

### 2. 工程师不会消失，但工程师的样子会变

行业不需要更多人写更多代码。需要的是能看到完整约束全景的工程师：
- 定义约束
- 设计测试
- 把控架构方向
- 判断什么问题值得解决

> **AI 负责实现，人负责方向。**

### 3. 品味才是最稀缺的能力

> **AI 可以写出任何你描述的东西，但不会告诉你该描述什么。代码越容易生成，品味就越值钱。**
