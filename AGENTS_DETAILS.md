## 规则1：消息接收去重检查

---
**rule_id**: RULE_001  
**name**: 消息接收去重检查  
**priority**: high  
**status**: active  
**created_at**: 2026-03-12  
**last_updated**: 2026-03-12  

---

### 规则描述

**触发条件**：收到用户消息时

**执行逻辑**：
```bash
python3 /root/.openclaw/workspace/queue_response_handler.py --check "用户输入"
```

| 返回结果 | 处理方式 |
|---------|---------|
| `is_response: true` | 立即按返回的 `message` 回复用户，不再执行其他逻辑 |
| `is_response: false` | 继续正常处理用户请求 |

**关键原则**：
- **先记录，后检查**：即使后续处理崩溃，指纹也已留存
- **永不跳过记录**：记录操作必须放在最外层 try-finally 中
- **12小时窗口**：覆盖飞书所有重试场景

**问题背景**：
Feishu 在消息处理超时会自动重试发送，导致 OpenClaw 收到重复消息，从而生成重复回复。

**正确做法**：
1. **收到消息的瞬间，立即记录指纹**（无论后续是否成功）
2. **然后检查是否是重复消息**：重复则回复 `NO_REPLY`
3. **继续正常处理消息**

### 规则溯源

**触发事件ID**: INC-2026-0312-001  
**事件时间**: 2026-03-12  
**事件类型**: 消息重复  
**严重级别**: P2

**现象描述**：
Feishu在消息处理超时会自动重试发送，OpenClaw收到重复消息后生成重复回复，导致用户收到多条相同回复。

**影响范围**：
- 用户体验下降
- 飞书渠道消息冗余

**根因分析**：
没有消息去重机制，无法识别重复消息。

**解决措施**：
增加规则1，通过消息指纹实现去重，12小时窗口覆盖飞书所有重试场景。

**验证结果**：
- 实施后重复消息事故：0次
- 去重成功率：100%

## 规则3：GitHub推送与凭据处理

---
**rule_id**: RULE_003  
**name**: GitHub推送与凭据处理  
**priority**: critical  
**status**: active  
**created_at**: 2026-03-10  
**last_updated**: 2026-03-20  

---

### 规则描述

**触发条件**：执行 git push 操作 或 处理凭据/Token/密码

**第一部分：凭据处理禁令（必执行）**

**严令禁止**：
- ❌ **删除任何凭据前必须先创建备份，否则无法恢复**
- ❌ **未经用户明确确认不得修改或删除凭据**
- ❌ **不得以"安全修复"为由擅自删除用户配置的 Token**

**必须执行**：
1. **备份优先**：修改前创建可恢复的备份，**至少保留1周**
2. **用户确认**：明确告知用户并获批准后方可操作
3. **功能验证**：修改后立即测试相关功能是否正常
4. **回滚方案**：确保有问题时可立即恢复原状

**事故案例**（2026-03-18）：
误将 GITHUB_TOKEN 识别为"硬编码密钥"并删除，未创建备份导致 Token 永久丢失。

**核心原则**：
> "任何修复都不能破坏原文件的运行逻辑，如果修复导致系统不能运行，那修复了还有什么意义。"

**第二部分：GitHub推送规范**

**推送前安全检查**：
```bash
# 四确认
echo "1. 当前目录: $(pwd)"
echo "2. Remote URL: $(git remote get-url origin)"
echo "3. 当前分支: $(git branch --show-current)"
echo "4. 嵌套仓库: $(find . -maxdepth 2 -name '.git' -type d)"

# 干跑验证
git push --dry-run origin 分支名
```

**推送方式**：
- **默认使用 SSH Key 模式**，不再询问使用 Personal Access Token 还是 SSH Key
- 确保 SSH 密钥已添加到 GitHub：`~/.ssh/id_ed25519.pub`
- 远程 URL 格式：`git@github.com:owner/repo.git`

**自动推送处理**：
- 检查通过后直接推送，**不询问用户**
- 失败自动重试（最多3次）
- 推送成功/失败告知用户结果

**禁止**：
- ❌ 询问"是否要推送"
- ❌ 询问"使用 Token 还是 SSH"
- ❌ 滥用 `--force`（必须使用 `--force-with-lease` 或先备份）
- ❌ 在错误目录执行推送

## 规则4：系统问题根因分析

---
**rule_id**: RULE_004  
**name**: 系统问题根因分析  
**priority**: high  
**status**: active  
**created_at**: 2026-03-19  
**last_updated**: 2026-03-19  

---

### 规则描述

**触发条件**：
- 系统定时监控发现异常
- 用户反馈定时任务、脚本或其他指令出现问题

**核心原则**：
> 不能应激性解决，不能就问题解决问题

**处理流程**：

```
发现问题
  ↓
【暂停】接到问题时，先不急于行动
  ↓
【收集】收集相关信息
  - 日志记录（最近执行日志）
  - 配置变更（git diff）
  - 依赖服务状态
  - 资源使用情况（CPU/内存/磁盘）
  ↓
【分析】从整个系统角度分析
  - 这是孤立问题还是系统性问题？
  - 最近有什么变更可能导致？
  - 问题的根本原因在哪里？
  ↓
【定位】确定解决问题的关键点
  - 关键问题是什么？
  - 解决优先级？
  - 影响范围评估？
  ↓
【建议】给出解决方案
  - 具体修复步骤
  - 验证方法
  - 回滚方案
  ↓
如果需要修改代码 ──是──→ 进入规则7（代码部署全流程）
  ↓ 否
执行修复并验证结果
```

**禁止行为**：
- ❌ 症状式修复（只修表面，不问根源）
- ❌ 未经分析直接套用过往经验
- ❌ 未定位关键点就动手修改

**示例对比**：
| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| "定时任务超时了，我把超时时间改大" | "定时任务超时是因为遍历了66个废弃文件，应该先清理文件再优化脚本" |
| "API调用失败，我重试几次" | "API调用失败是因为用了错误的API Key，应该切换到正确的认证方式" |

**输出要求**：
分析完成后必须向用户说明：
1. 问题根本原因
2. 关键解决点在哪里
3. 建议的修复方案
4. 是否需要进入规则7进行代码部署

### 规则溯源

## 规则5：定时任务配置与监控

---

**priority**: critical

---

### 规则描述

**触发条件**：新增、修改定时任务，或定时任务监控告警

**任务分类与处理策略**：

| 任务类型 | 定义 | 用户干扰策略 |
|---------|------|-------------|
| **静默任务** | 后台运行，不向用户主动发送消息的任务 | **无故障不打扰**：后台自动运行，仅在失败时告警 |
| **主动发送任务** | 需要向用户主动发送消息的任务（如飞书推送） | **必须验证**：新增/修改时强制端对端模拟验证 |

**主动发送任务 - 强制验证流程**

**1. 引用通用检查清单**（第2章）
- [ ] **环境确认**
- [ ] **风险评估**
- [ ] **备份创建**
- [ ] **依赖检查**
- [ ] **测试验证**（差异化：必须dry-run）
- [ ] **监控确认**

**2. 端对端模拟验证（差异化要求 - 必须）**

新增或修改主动发送任务前，必须执行：
```bash
# 模拟任务执行（不实际发送，验证流程完整性）
python3 /path/to/task_script.py --dry-run --verbose

# 检查输出是否符合预期
echo $?  # 必须为 0
```

**验证内容**：
- [ ] 任务能正常启动
- [ ] 消息内容生成正确
- [ ] 发送逻辑路径完整
- [ ] 错误处理机制有效

**3. 沙箱测试（推荐）**

对于重要任务，先在小范围测试：
```bash
# 使用测试频道/测试用户
python3 task_script.py --target=test --notify=false
```

**任务运行监控**

**监控范围**：
- 任务执行状态（成功/失败）
- 执行耗时异常（超过历史平均 200%）
- 输出结果为空或异常
- 消息发送链路中断

**监控方式**：
```bash
# 每小时健康检查（静默模式）
python3 /root/.openclaw/skills/cron-health-dashboard/scripts/cron_health_check.py --silent
```

**故障处理与AI分析**

**触发AI分析的条件**：
1. 监控发现任务失败
2. 其他代码修改导致任务运行异常
3. 定时发送链路中断

**AI分析流程**：

**步骤1：信息收集**
```bash
# 收集任务相关信息
- 最近执行日志
- 相关代码变更（git diff）
- 依赖服务状态
```

**步骤2：调用AI分析**
- **首选模型**：`qwen/qwen3.5-coder-plus`（分析速度快）
- **失败切换**：超时或失败 → `kimi-coding/k2p5`
- **调用方式**：`sessions_spawn` 启动子 Agent

**分析内容**：
```markdown
请分析定时任务故障：

**任务信息**：
- 任务名称：xxx
- 最后执行时间：xxx
```

## 规则7：代码部署全流程

---
**rule_id**: RULE_007  
**name**: 代码部署全流程  
**priority**: critical  
**status**: active  
**created_at**: 2026-03-05  
**last_updated**: 2026-03-19  

---

### 规则描述

**触发条件**：修改代码或新建代码文件、修复代码后部署  
**前置条件**：已完成规则4根因分析（如适用），问题已定位清楚

**核心原则**：无测试、无审计、无备份，禁止部署

**阶段一：部署前强制检查**

**1. 引用通用检查清单**（第2章）
- [ ] **环境确认** - 当前目录、远程地址、分支正确
- [ ] **风险评估** - 操作影响范围、回滚方案
- [ ] **备份创建**（强制）
  ```bash
  git add .
  git commit -m "backup: 部署前备份 $(date '+%Y-%m-%d %H:%M:%S')"
  git branch backup-$(date '+%Y%m%d-%H%M%S')
  ```
- [ ] **依赖检查** - 必要服务、API、权限正常
- [ ] **测试验证** - dry-run、单元测试、集成测试
- [ ] **监控确认** - 日志记录、告警机制就绪

**2. 影响范围检查（新增 - 差异化要求）**

**触发条件**：修改或删除代码/文件时

**检查内容**：
```bash
# 1. 查找被修改文件的引用者
grep -r "from.*modified_file" --include="*.py" .
grep -r "import.*modified_file" --include="*.py" .

# 2. 查找被删除文件的引用者
grep -r "deleted_file" --include="*.py" .

# 3. 查找函数/类的调用者
grep -r "function_name(" --include="*.py" .
```

**强制要求**：
- [ ] **修改代码前**：确认所有引用该代码的模块
- [ ] **删除文件前**：确认无任何模块引用该文件
- [ ] **重命名前**：确认所有import路径需要同步更新
- [ ] **评估影响**：向用户说明修改将影响哪些模块

**禁止**：
- ❌ 未检查引用关系直接修改核心代码
- ❌ 删除文件前未确认是否有其他文件import
- ❌ 修改公共API前未检查调用方兼容性

**事故案例**（2026-03-23）：
删除文件时未检查引用关系，导致依赖该文件的其他模块报错，系统功能异常。

**3. AI 深度审计（差异化要求）**

**触发条件**：修改/新增代码文件超过 50 行，或涉及核心逻辑变更

**审计方式**：
- **首选模型**：`alibaba/qwen3.5-plus`（速度快、成本低）
- **失败切换**：如果 Qwen3.5 失败或超时，自动切换到 `kimi-coding/k2p5`
- **调用方式**：通过 `sessions_spawn` 启动子 Agent 进行异步审计

**审计内容**：
- 代码逻辑是否合理，有无明显 bug
- 边界情况是否处理（空值、超大文件、网络失败）
- 错误处理是否完善（异常捕获、回滚机制）
- 性能是否有问题（循环、递归、资源泄漏）
- 是否符合项目代码风格

**审计结果处理**：
- 【PASS】→ 继续部署流程
- 【FAIL】→ 必须修复问题后才能部署

**阶段二：部署执行**

- 确认文件部署到正确位置
- 检查文件权限
- 验证配置已更新
- 关键路径必须有日志记录

**阶段三：部署后验证与保障**

- 部署后立即测试核心功能
- 等待定时任务实际执行（如相关）
- 检查执行结果

**自动回滚机制**：
```bash
# 如果部署后测试失败，自动回滚
git reset --hard backup-branch
git push origin main --force-with-lease
```

**失败报告**：
- 部署或运行失败立即记录到 `.learnings/DEPLOY_FAILURES.md`
- 包含：时间、组件、错误信息、根因分析、修复方案、预防措施
- 通知用户失败详情

**禁止事项**：
- ❌ 无测试直接部署
- ❌ 无审计直接部署
- ❌ 无备份直接部署
- ❌ 部署失败不报告
- ❌ 声称"已修复"直到验证通过

### 规则描述

**触发条件**：
- 复杂问题需要多角度分析
- 任务可以分解为多个独立子任务

**可用模型及优势**（阿里百炼coding plan）：

| 模型 | 优势场景 | 分配策略 |
|-----|---------|---------|
| **GLM-5** | 中文理解、逻辑推理 | 分析类任务、根因定位 |
| **MiniMax-M2.5** | 长上下文、代码生成 | 代码审查、文档生成 |
| **Qwen3.5 Plus** | 速度、成本效益 | 快速验证、简单查询 |
| **k2p5** | 综合能力、高准确率 | 最终审核、关键决策 |

**分配原则**：

```
复杂问题
  ↓
【任务分解】拆分为独立子任务
  - 子任务A：架构分析
  - 子任务B：代码审查
  - 子任务C：快速验证
  ↓
【模型匹配】根据特性分配最适合的模型
  ↓
【并行执行】同时启动多个子Agent
  ↓
【结果整合】综合各子Agent输出，形成最终结论
```

**调用方式**：
```bash
# 示例：并行分析代码问题
openclaw sessions spawn \
  --task "分析系统架构问题" \
  --model "zhipu/glm-5" \
  --label arch-analysis &

openclaw sessions spawn \
  --task "审查代码实现细节" \
  --model "minimax/minimax-m2.5" \
  --label code-review &

openclaw sessions spawn \
  --task "快速验证语法正确性" \
  --model "alibaba/qwen3.5-plus" \
  --label syntax-check &

wait
```

**成本控制**：
- 简单任务优先用Qwen3.5 Plus
- 复杂任务用GLM-5或MiniMax-M2.5
- 关键决策必须用k2p5审核

**结果整合**：
- 汇总各子Agent输出
- 识别共识和分歧
- 综合形成最终建议
- 必要时用k2p5做最终审核
