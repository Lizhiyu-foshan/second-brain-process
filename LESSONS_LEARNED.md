## 质量保障机制

### 1. 部署前强制检查

**执行脚本**：`./deploy_check.sh`

检查项：
- [ ] 语法检查通过
- [ ] 功能测试通过
- [ ] 性能测试通过（<20秒）
- [ ] 配置检查通过
- [ ] 大文件检查通过

**规则**：检查不通过，禁止部署

### 2. 代码审查清单

**修改定时任务代码前，必须自问**：

```
□ 是否考虑了大数据量场景？
□ 是否有超时保护？
□ 循环逻辑是否可能无限循环？
□ 文件IO是否有大小限制？
□ 子Agent是否真正执行而非模拟？
□ 是否有日志记录便于排查？
```

### 3. 测试流程

**本地测试** → **代码审查** → **部署检查** → **上线**

**本地测试命令**：
```bash
# 1. 语法检查
python3 -m py_compile script.py

# 2. 功能测试
timeout 10 python3 script.py --test

# 3. 性能测试
time python3 script.py

# 4. 完整部署检查
./deploy_check.sh
```

### 4. 监控与告警

**已配置**：
- Cron任务健康监控（每15分钟）
- 复盘报告补发机制
- 消息送达校验

**待完善**：

## 经验教训

### 关于Agent幻觉

**问题**：子Agent会生成看起来像执行结果的文本，而非真正执行

**对策**：
1. 使用独立的Python脚本封装实际逻辑
2. 子Agent只负责调用脚本，不处理业务逻辑
3. 脚本必须有明确的退出码和日志

### 关于性能

**问题**：开发时只考虑小数据量，未测试大数据量场景

**对策**：
1. 所有文件操作必须限制大小
2. 所有循环必须有最大迭代次数
3. 所有时间窗口必须有最大回溯限制

### 关于测试

**问题**：修复后直接上线，未做回归测试

**对策**：
1. 建立deploy_check.sh强制检查
2. 修改后必须运行测试才能提交
3. 关键路径必须有单元测试
