## 🆕 新增环节：代码审查 (Stage 5)

### 5.1 代码审查Agent职责

```
┌─────────────────────────────────────────┐
│         👤 代码审查员 CodeReviewer       │
├─────────────────────────────────────────┤
│                                         │
│  输入: 开发者提交的代码                   │
│                                         │
│  审查维度:                               │
│  ├── 🔍 代码规范 (PEP8, 命名规范)        │
│  ├── 🔍 安全漏洞 (SQL注入, XSS等)        │
│  ├── 🔍 性能问题 (N+1查询, 内存泄漏)     │
│  ├── 🔍 设计模式 (SOLID原则)             │
│  ├── 🔍 代码复杂度 (圈复杂度)            │
│  ├── 🔍 测试覆盖 (单元测试完整性)        │
│  └── 🔍 文档完整性 (注释, docstring)     │
│                                         │
│  输出:                                   │
│  ├── 📄 代码审查报告                      │
│  ├── ⚠️  问题清单 (按严重程度分类)        │
│  └── 💡 改进建议                          │
│                                         │
└─────────────────────────────────────────┘
```

### 5.2 代码审查检查清单

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| **代码规范** | P0 | PEP8, 命名规范, 代码格式 |
| **安全漏洞** | P0 | SQL注入, XSS, 敏感信息泄露 |
| **性能问题** | P1 | N+1查询, 循环内查询, 内存泄漏 |
| **异常处理** | P1 | 异常捕获, 错误处理, 日志记录 |
| **设计模式** | P1 | SOLID原则, 代码复用, 解耦 |
| **测试覆盖** | P2 | 单元测试, 边界条件, 异常场景 |
| **文档注释** | P2 | 函数注释, 复杂逻辑说明 |

### 5.3 代码审查报告示例

```markdown
# 代码审查报告

**审查对象**: services/cart_service.py  
**审查时间**: 2026-02-21  
**审查员**: CodeReviewer  
**代码作者**: Amelia

## 审查结果

| 检查项 | 状态 | 问题数 | 严重程度 |
|--------|------|--------|----------|
| 代码规范 | ⚠️ 警告 | 3 | 低 |
| 安全漏洞 | ✅ 通过 | 0 | - |
| 性能问题 | ⚠️ 警告 | 1 | 中 |
| 异常处理 | ❌ 失败 | 2 | 高 |
| 设计模式 | ✅ 通过 | 0 | - |
| 测试覆盖 | ⚠️ 警告 | 1 | 中 |

## 关键问题

### 🔴 高优先级 (必须修复)

1. **异常处理不完善**
   - 位置: line 45, `add_to_cart()`
   - 问题: 未处理数据库连接异常
   - 建议: 添加 try-except 块，记录错误日志

2. **事务管理缺失**
   - 位置: line 78, `update_cart_item()`
   - 问题: 库存检查和扣减不在同一事务
   - 建议: 使用 @transactional 装饰器

## 改进建议

1. 添加函数文档字符串
2. 优化数据库查询 (使用 joinedload)
3. 增加单元测试覆盖率

## 审查结论

**状态**: ❌ **不通过** (存在高优先级问题)

**建议**: 修复高优先级问题后重新提交审查
```

## 🛡️ 代码审查的关键作用

### 1. 质量门禁

```
开发完成
    │
    ▼
┌─────────────┐
│ 代码审查    │ ← 质量门禁
│ CodeReview  │
└──────┬──────┘
       │
    ┌──┴──┐
    │通过? │
    └──┬──┘
       │
   是  │  否
   ┌───┘    └───┐
   ▼            ▼
进入测试    返回修复
```

**作用**: 在测试前拦截代码质量问题，减少测试阶段缺陷

### 2. 知识共享

- 审查过程 = 团队学习过程
- 发现最佳实践
- 统一代码风格
- 传播设计模式

### 3. 风险早期发现

| 风险类型 | 发现阶段 | 修复成本 |
|---------|---------|---------|
| 安全漏洞 | 代码审查 | 低 |
| 性能问题 | 代码审查 | 低 |
| 设计缺陷 | 代码审查 | 中 |
| 功能缺陷 | 测试阶段 | 高 |
| 生产故障 | 上线后 | 极高 |

**价值**: 早期发现问题，降低修复成本 10-100倍
