# 自我进化机制对比分析

## 核心观点

对比分析了两种AI自我进化机制：**self-improving-agent 技能的自动捕获机制**与**本系统的人工反馈+规则固化机制**，两者各有优势，可以相互补充。

## 关键讨论

### 1. self-improving-agent 机制

**核心功能：**
- 📝 记住用户的纠正，不再犯同样的错误
- 🐛 自动记录失败的操作和解决方案
- 💡 积累最佳实践，提升工作效率
- 🔄 持续改进，越用越聪明

**三层日志系统：**

| 文件 | 用途 | 触发条件 |
|------|------|----------|
| `LEARNINGS.md` | 学习记录、用户纠正、知识更新 | 用户纠正、发现更好方法 |
| `ERRORS.md` | 命令失败、API错误、异常情况 | 操作失败、API错误 |
| `FEATURE_REQUESTS.md` | 用户请求但暂不支持的功能 | 用户提新需求 |

**自动触发条件：**
```
❌ 命令/操作意外失败
🔄 用户纠正 AI（"不对，应该是..."）
🆕 用户请求不存在的功能
⚠️ 外部 API 或工具失败
📚 发现知识过时或错误
✨ 发现更好的解决方法
```

**学习推广机制：**
| 学习类型 | 推广目标 | 示例 |
|----------|----------|------|
| 行为准则 | `SOUL.md` | "简洁回复，避免废话" |
| 工作流程 | `AGENTS.md` | "长任务派生子智能体" |
| 工具使用 | `TOOLS.md` | "Git push 需要先配置认证" |
| 项目知识 | `MEMORY.md` | 项目特定约定和配置 |

### 2. 本系统的规则固化机制

**核心原则：**
- 犯错→反馈→写规则→不犯
- 人工确认后才推广到长期记忆
- 迭代计数器：每次对话后问"这次能固化什么？"

**规则格式（AGENTS.md）：**
```markdown
### 规则X：XXX（YYYY-MM-DD）
**Priority**: high/medium/low
**Status**: active/deprecated/pending-review

**触发条件：** XXX
**错误案例：** XXX
**正确做法：** XXX
**检查方式：** XXX
```

**双层记录体系：**
```
发现 → 记录到 .learnings/ → 验证（迭代计数器）→ 推广到长期记忆
     ↑                                              ↓
     └──────────── 定期复盘（Heartbeat）────────────┘
```

### 3. 两种机制对比

| 维度 | self-improving-agent | 本系统 |
|------|---------------------|--------|
| **捕获方式** | 自动捕获 | 人工反馈 |
| **验证机制** | 自动推广 | 人工确认后推广 |
| **粒度** | 细粒度（每次错误） | 粗粒度（规则级别） |
| **可控性** | 低（自动） | 高（人工审核） |
| **遗漏风险** | 低 | 中（依赖人工发现） |
| **错误积累** | 可能（自动推广错误） | 低（人工过滤） |

### 4. 融合方案

**结合两者的优势：**

```
┌─────────────────────────────────────────┐
│         L1 自动捕获层                   │
│  （借鉴 self-improving-agent）          │
│  - 自动记录所有错误                     │
│  - 自动记录用户纠正                     │
│  - 自动记录功能请求                     │
└─────────────────┬───────────────────────┘
                  ↓ 每日复盘时人工审查
┌─────────────────────────────────────────┐
│         L2 人工验证层                   │
│  （本系统现有机制）                      │
│  - 审查自动捕获的内容                   │
│  - 判断是否有广泛适用性                 │
│  - 决定推广到长期记忆                   │
└─────────────────┬───────────────────────┘
                  ↓ 验证有效后
┌─────────────────────────────────────────┐
│         L3 固化规则层                   │
│  - AGENTS.md（工作流程）                │
│  - SOUL.md（行为准则）                  │
│  - TOOLS.md（工具使用）                 │
│  - MEMORY.md（项目知识）                │
└─────────────────────────────────────────┘
```

## 有价值的思考

1. **自动 vs 人工的平衡**
   - 自动捕获确保不遗漏
   - 人工验证确保质量
   - 两者的结合是最佳状态

2. **规则的生命力**
   - 规则需要定期review（Heartbeat机制）
   - 过时的规则要标记为deprecated
   - 新规则要标注pending-review状态

3. **迭代计数器的重要性**
   - 不只是记录改进了什么
   - 更要记录改进了多少次
   - 迭代次数与产出质量正相关

## 可能的改进

1. **引入 self-improving-agent 的自动捕获**
   - 在当前系统基础上增加 `.learnings/` 目录
   - 自动捕获所有错误和纠正
   - 每日复盘时审查并决定是否推广

2. **推广机制自动化**
   - 当某个学习被验证有效N次后
   - 自动提示用户是否推广到长期记忆
   - 减少人工判断负担

3. **学习效果的量化追踪**
   - 记录每个规则的应用次数
   - 统计规则避免的重复错误数
   - 用数据验证规则的有效性
