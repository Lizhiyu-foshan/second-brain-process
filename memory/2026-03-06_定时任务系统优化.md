# 定时任务系统优化与代码质量保障

## 核心观点

构建了一个完善的**定时任务代码质量保障体系**，通过部署前检查、性能优化、安全机制三层防护，确保凌晨5:00整理任务的稳定运行。

## 关键讨论

### 1. 时间窗口逻辑修复

**问题发现：**
- 原逻辑只处理"上次处理时间到现在"，无法正确处理"前一天5:00到今天5:00"
- 用户明确指出：要求的是处理前一天5:00到今天5:00的笔记和聊天记录

**解决方案：**
```python
# 新增 get_fixed_daily_window() 函数
def get_fixed_daily_window():
    """获取固定的24小时窗口（昨天5:00到今天5:00）"""
    now = datetime.now()
    today_5am = now.replace(hour=5, minute=0, second=0, microsecond=0)
    
    # 如果现在是5:00之前，昨天5:00是起始时间
    if now < today_5am:
        start_time = today_5am - timedelta(days=1)
        end_time = today_5am
    else:
        # 现在已过5:00，今天5:00是起始时间
        start_time = today_5am
        end_time = today_5am + timedelta(days=1)
    
    return start_time, end_time
```

### 3. 文章截图删除逻辑优化

**原问题：**
- 只按文件大小判断（>1MB就删）
- 没有检查是否真正提取了信息

**新逻辑（三条件判断）：**
```
条件1：文件 > 1MB（可能是截图）
    ↓
条件2：是文章截图（文件名含关键词 或 内容含文章特征）
    ↓
条件3：信息已提取（通过URL hash检查对应笔记是否存在）
    ↓
满足三个条件 → 安全删除
```

### 4. 性能与安全优化

**文件处理限制：**
- 最大文件大小：2MB
- 最多解析行数：10,000行
- 最多处理文件数：15个
- 最大回溯时间：24小时

**文件锁机制：**
```python
# 使用Linux flock防止并发执行
def acquire_lock():
```

## 有价值的思考

1. **检查清单的力量**
   - 代码审查前必须运行 `./deploy_check.sh`
   - 所有检查通过才能部署
   - 失败必须记录到 `.learnings/DEPLOY_FAILURES.md`

2. **防御性编程**
   - 限制文件大小和解析行数
   - 所有循环必须有最大迭代次数
   - 子Agent只调用封装脚本，不直接处理业务逻辑
   - 关键路径必须有日志记录

3. **时间窗口的精确性**
   - 不要依赖"上次处理时间"
   - 使用固定时间窗口确保不遗漏
   - 考虑边界情况（跨天、时区）
