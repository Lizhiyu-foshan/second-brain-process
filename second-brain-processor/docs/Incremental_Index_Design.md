# 增量消息索引系统设计方案

> 解决会话文件分散、mtime不可靠导致的扫描性能问题
> 版本: v1.0
> 日期: 2026-03-21

---

## 一、问题根因分析

### 当前系统的问题

```
用户对话 → OpenClaw创建会话 → 写入 {session_id}.jsonl
      ↓                              ↓
  同一会话持续对话                    文件句柄保持打开
      ↓                              ↓
  跨天消息产生                        mtime不更新
      ↓                              ↓
  昨天的消息在mtime=3天前的文件中      按mtime筛选会漏扫
```

### 数据验证

| 扫描策略 | 扫描文件数 | 提取消息数 | 漏扫率 |
|---------|-----------|-----------|--------|
| 按mtime近1天 | 50 | 1307 | **57%** |
| 按mtime近3天 | 163 | 1401 | 0% |
| 全量扫描 | 163 | 1401 | 0% |

**结论**: mtime不可靠，必须全量扫描，但随着文件增长会再次超时。

---

## 二、方案A：增量索引系统架构

### 核心思想

不再依赖文件系统的时间属性，而是维护一个**消息级别的时间索引**，记录每次处理到的最后消息时间戳，下次从该时间点继续处理。

```
Day 1 (2026-03-20):
  处理消息 timestamp: 1000-2000
  记录索引: last_timestamp = 2000

Day 2 (2026-03-21):
  扫描所有文件，只提取 timestamp > 2000 的消息
  处理消息 timestamp: 2001-3500
  更新索引: last_timestamp = 3500

Day 3 (2026-03-22):
  只提取 timestamp > 3500 的消息
  ...
```

#### 2. Scanner（扫描器）

**优化策略**:
```python
def scan_incrementally(last_timestamp):
    """
    增量扫描策略
    1. 遍历所有会话文件（不关心mtime）
    2. 对每个文件，快速定位到 > last_timestamp 的消息
    3. 只读取和处理新增消息
    """
    messages = []
    
    for jsonl_file in get_all_session_files():
        # 快速检查：读取文件最后几行判断是否有新消息
        if not has_new_messages(jsonl_file, last_timestamp):
            continue
            
        # 只读取新增部分
        new_messages = read_messages_after(jsonl_file, last_timestamp)
        messages.extend(new_messages)
    
    return messages
```

**关键优化**:
- **尾部检查**：读取文件最后5行，如果有消息 > last_timestamp，说明文件有更新
- **快速跳过**：如果最后修改时间 < last_timestamp，直接跳过整个文件
- **内存效率**：不一次性读取整个文件，只读取需要的部分

#### 3. Writer（写入器）

**职责**:
- 按日期聚合消息
- 生成 Markdown 文件
- 更新索引

## 三、处理流程

### 完整流程图

```
每天 5:00 定时任务触发
        ↓
┌──────────────────┐
│ 1. 读取索引       │
│ 获取 last_timestamp│
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 2. 增量扫描       │
│ 只读取 > last_ts  │
│ 的消息           │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 3. 处理消息       │
│ 去重、格式化     │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 4. 生成文件       │
│ 保存到 Obsidian  │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 5. 更新索引       │
│ 记录新的 last_ts  │
│ 和统计数据        │
└────────┬─────────┘
         ↓
    完成 ✅
```

### 边界情况处理

| 场景 | 处理策略 |
|------|---------|
| 首次运行（无索引） | 全量扫描，建立初始索引 |
| 索引文件损坏 | 自动重建索引（全量扫描） |
| 系统时间回拨 | 检测并报警，使用最新有效时间 |
| 消息时间戳乱序 | 按时间戳排序处理，不依赖文件顺序 |
| 文件被删除 | 忽略，不影响索引 |

## 四、性能分析

### 复杂度对比

| 指标 | 原方案（3天扫描） | 增量索引方案 |
|------|-----------------|-------------|
| 时间复杂度 | O(N_files × N_lines) | O(N_files_with_new_msg × N_new_lines) |
| 文件数量增长影响 | 线性恶化 | 基本无关 |
| 消息数量增长影响 | 线性增长 | 只处理新增 |
| 预期执行时间 | 0.3-50秒（随文件增长） | **<0.1秒（稳定）** |

### 性能预估

假设：
- 每天新增会话文件：5-10个
- 每天新增消息：500-2000条
- 总会话文件：持续增长到1000+

| 运行日 | 原方案 | 增量方案 |
|--------|--------|---------|
| Day 1 | 0.3秒 | 0.3秒（首次全量） |
| Day 30 | 1.5秒 | 0.05秒 |
| Day 90 | 5秒 | 0.05秒 |
| Day 365 | 可能超时 | 0.05秒 |
