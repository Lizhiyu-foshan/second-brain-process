## 一、时间窗口定义

### 核心逻辑：固定24小时窗口
```python
# 处理范围：前一天5:00 到 今天5:00
# 示例：2026-03-07 05:00 执行时，处理 2026-03-06 05:00 至 2026-03-07 05:00 的内容

yesterday_5am = today 05:00 - 1 day
today_5am = today 05:00
window_size = 24 hours (fixed)
```

### 为什么用固定窗口？
- **可靠性**：不受上次处理时间影响，即使某天任务失败也不会遗漏内容
- **可预测**：每天处理的内容范围明确，便于调试和验证
- **防重复**：天然防止同一内容被多次处理

## 三、步骤2详细：内容整理逻辑

### 3.1 数据来源
```python
SESSION_DIR = "/root/.openclaw/agents/main/sessions"
# 文件格式: {uuid}.jsonl (每行一个JSON对象)
```

### 3.2 文件过滤规则
| 条件 | 处理 | 说明 |
|------|------|------|
| 修改时间 < 昨天5:00 | 跳过 | 不在处理窗口内 |
| 修改时间 > 今天5:00 | 跳过 | 不在处理窗口内 |
| 文件大小 > 2MB | 跳过 | 可能是截图文章，单独处理 |
| 文件数量 > 15个 | 停止 | 防止处理过多文件 |

### 3.3 内容分类（A/B/C模型）

```
解析会话文件
     │
     ├───→ A类：独立对话 ───→ 直接保存为笔记
     │       (无URL，纯对话)
     │
     ├───→ B类：链接+讨论 ───→ 附加到链接笔记
     │       (有URL，有我的参与讨论)
     │
     ├───→ C类：链接+AI整理 ──→ 提取关键观点
     │       (有URL，AI自动整理)
     │
     └───→ 待处理链接 ───→ 加入待处理队列
             (有URL，无讨论)
```

#### A类：独立对话
- **识别条件**：消息中无URL，且内容长度>20字符
- **处理方式**：
  - 生成文件名：`{日期}_{标题前30字}.md`
  - 保存位置：`obsidian-vault/02-Conversations/`
  - 文件格式：Markdown with YAML frontmatter
  - Frontmatter包含：date, type, tags

#### B类：链接+我的讨论
- **识别条件**：
  - 消息中包含URL
  - 后续有我参与的长文本讨论（>50字符）
- **处理方式**：
  - 提取URL作为笔记主键
  - 讨论内容附加到该链接笔记
  - 触发飞书提醒（可选）

#### C类：链接+AI自动整理
- **识别条件**：
  - 消息中包含URL
  - 有AI整理输出，但无我的深度参与
- **处理方式**：
  - 提取关键观点
  - 生成精简摘要笔记
  - 保存到 `obsidian-vault/03-Articles/`

### 3.4 文件命名规范
```
{YYYYMMDD}_{标题}_{序号}.md

示例：
- 2026-03-06_凌晨5点聊天记录整理.md
- 2026-03-06_AI边界讨论_1.md
- 2026-03-06_复盘报告推送_2.md
```

### 3.5 GitHub推送
```
整理完成
    │
    ├───→ git add .
    ├───→ git commit -m "auto: 凌晨整理 - 保存 X 条对话记录"
    └───→ git push
    
失败处理：
    └───→ 记录到 ERRORS.md
    └───→ 下次执行时补推
```

## 七、性能保障

### 7.1 资源限制
| 项目 | 限制 | 目的 |
|------|------|------|
| 单个文件大小 | 2MB | 防止大文件卡住 |
| 单次处理文件数 | 15个 | 控制总耗时 |
| 解析行数 | 10,000行 | 防止内存溢出 |
| 时间窗口 | 24小时 | 防止处理过多历史内容 |

### 7.2 并发控制
- 使用Linux flock实现文件锁
- 防止多个实例同时运行
- 锁文件：`/tmp/kimiclaw_v2.lock`

### 7.3 超时处理
- Git操作：30-60秒超时
- 文件读取：内置限制
- 整体任务：300秒超时
