# 动态上下文压缩守护

## 问题背景

长时间对话后，会话上下文不断增长：
- 达到模型限制（如 k2p5 的 256k tokens）
- 需要手动执行 `/compact` 指令
- 容易忘记，导致通信中断

## 解决方案

**完全自动化的上下文压缩系统**

### 核心机制

```
每5分钟检查一次会话状态
         ↓
┌─────────────────────────────┐
│ 检查消息数 >= 100条？      │──是──┐
└─────────────────────────────┘      │
         否                          │
         ↓                           │
┌─────────────────────────────┐      │
│ 检查 Token >= 25万？       │──是──┤
└─────────────────────────────┘      │
         否                          │
         ↓                           │
      正常，跳过                     ↓
                              触发自动压缩
                                    ↓
                              调用 openclaw compact
                                    ↓
                              发送通知
```

### 双重阈值

| 阈值类型 | 数值 | 说明 |
|---------|------|------|
| **消息数** | 100条 | 高频对话场景 |
| **Token 软限制** | 20万 | 提前预警 |
| **Token 硬限制** | 25万 | 紧急压缩 |
| **冷却时间** | 30分钟 | 避免频繁压缩 |

### Token 估算

```python
# 中文字符按2 tokens，英文按1 token
tokens = chinese_chars * 2 + other_chars
```

## 工作原理

### 与其他后台检查同步

所有后台检查任务统一配置：

| 任务 | 频率 | 模式 | 状态 |
|------|------|------|------|
| 消息延迟监控 | 每10分钟 | 静默，异常通知 | ✅ 运行中 |
| 文章剪藏健康检查 | 每10分钟 | 静默，异常通知 | ✅ 运行中 |
| **上下文压缩** | 每10分钟 | **静默，异常通知** | ✅ **已同步** |

### 1. 监控指标

| 指标 | 监控方式 | 触发条件 |
|------|---------|---------|
| 消息行数 | 统计 .jsonl 文件行数 | >= 100 |
| Token 数 | 估算字符数 | >= 20万软限制 / 25万硬限制 |
| 冷却时间 | 记录上次压缩时间 | >= 30分钟 |

### 2. 压缩触发

当满足以下任一条件时触发：
- ✅ 消息数 >= 100 条
- ✅ Token 数 >= 20万（软限制）
- ✅ Token 数 >= 25万（硬限制）

**且** 冷却期已过（30分钟）

### 3. 压缩执行

```python
# 方式1: 创建压缩标记文件
# 检测到需要压缩时创建标记

# 方式2: 通知用户手动执行 /compact
# 发送飞书通知提醒用户
```

**注意**：由于 OpenClaw 没有直接的 compact API，系统会：
1. 创建压缩标记文件
2. 发送飞书通知提醒用户执行 `/compact`
3. 用户执行后，标记自动清除

### 4. 通知策略

**静默模式**（默认）：
- ✅ 正常压缩 → 仅记录日志，**不发送飞书通知**
- ⚠️ 压缩失败 → 发送告警通知
- 🔴 达到硬限制 → 发送紧急通知

**通知内容**：
- 压缩原因
- 当前统计
- 累计节省 tokens
- 建议操作（如需要）

## 与其他压缩机制对比

| 机制 | 触发方式 | 优点 | 缺点 |
|------|---------|------|------|
| **手动 /compact** | 人工触发 | 完全控制 | 容易忘记 |
| **auto-fix 定时压缩** | 基于文件大小 | 后台静默 | 不够智能 |
| **本方案（动态压缩）** | 基于消息/token | 精准触发、完全自动 | 需要估算token |

## 故障排查

### 问题：没有自动压缩

**检查清单：**
1. 定时任务是否正常运行？
2. 冷却期是否已过？
3. 是否达到阈值？

**调试：**
```bash
# 查看日志
tail -20 /root/.openclaw/workspace/.learnings/context_monitor.log

# 手动运行检查
python3 dynamic_compactor.py --check
```

### 问题：频繁压缩

**原因：**
- 阈值设置过低
- 冷却时间太短

**解决：**
```python
# 提高阈值
MSG_COUNT_THRESHOLD = 200        # 提高到200条
CONTEXT_SOFT_LIMIT = 220000      # 提高到22万
COOLDOWN_MINUTES = 60            # 冷却1小时
```

## 最佳实践

1. **保持默认配置**：已针对 k2p5 (256k) 优化
2. **监控日志**：定期检查压缩频率
3. **配合手动压缩**：特殊情况仍可使用 `/compact`
4. **关注通知**：及时了解压缩情况
