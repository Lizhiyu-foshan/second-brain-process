# 修复验证清单 (Fix Verification Checklist)

**目的**：确保所有修复都经过充分验证，避免"修复 - 复发"循环和引入新问题

---

## 验证流程（必须完成所有步骤）

### 步骤 1: 修复代码
- [ ] 理解问题的完整上下文
- [ ] 进行 5 Why 根因分析
- [ ] 编写修复代码
- [ ] 代码审查（边界情况、失败场景）

### 步骤 2: 单元测试
```bash
# 示例命令
cd /root/.openclaw/workspace/second-brain-processor
python3 -c "
from <module> import <function>
result = <function>('测试')
print('✅ 单元测试通过')
"
```
- [ ] 函数可以正常导入
- [ ] 函数签名正确
- [ ] 基础调用不报错
- [ ] 返回数据结构符合预期

### 步骤 3: 集成测试
```bash
# 模拟实际调用场景
python3 run_morning_process_progress.py --test 2>&1 | tail -50
```
- [ ] 模拟实际调用场景
- [ ] 测试边界条件
- [ ] 验证与其他模块的交互
- [ ] 检查日志无错误

### 步骤 4: 完整流程验证（关键！）
```bash
# 手动触发一次完整流程
python3 run_morning_process_progress.py 2>&1 | tail -50

# 检查输出文件
ls -la /root/.openclaw/workspace/memory/ | tail -5
cat /root/.openclaw/workspace/memory/2026-03-16.md | tail -20
```
- [ ] 手动触发一次完整任务流程
- [ ] 验证任务输出符合预期
- [ ] 检查日志无错误
- [ ] 检查文件时间戳更新

### 步骤 5: 推送代码
```bash
cd /root/.openclaw/workspace
git add .
git commit -m "fix: <描述>

- 根因：<5 Why 分析结果>
- 修复：<具体修改>
- 验证：<验证步骤>
"
git push
```
- [ ] git add / commit / push
- [ ] 确认远程代码已更新
- [ ] 提交信息包含验证结果

### 步骤 6: 下次任务确认（最关键！）
- [ ] 如果是定时任务，等待下次执行
- [ ] 检查执行结果日志
- [ ] 验证输出文件已更新
- [ ] 向用户报告"已修复并验证，下次任务执行时间是 X 点"

---

## 禁止事项

❌ 修复后只做简单导入测试就声称"已修复"
❌ 不验证完整流程就等待定时任务自动执行
❌ 不对修复结果进行追踪确认
❌ 用户问起来才说"可能还没生效"

---

## 承诺规范

❌ 错误："明天应该能正常工作"
✅ 正确："已修复并验证，下次任务执行时间是 X 点，我会检查执行结果并向您汇报"

---

## 失败处理

如果验证失败：
1. 立即回滚或继续修复
2. 更新错误记录到 `.learnings/ERRORS.md`
3. 向用户坦诚说明"验证未通过，继续修复中"
4. 不得声称"已修复"直到验证通过
