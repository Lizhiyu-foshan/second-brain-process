# 对话整理工作流程规则 v2.0

> 两阶段整理 + 单次推送原则

---

## 核心原则

**GitHub 仓库只保存最终版本，不保留中间迭代。**

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  阶段1: 初步整理  │────→│  阶段2: AI整理   │────→│  阶段3: 推送到GitHub │
│  (本地完整保存)  │     │  (主题精炼)     │     │  (最终版本)     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
      5:00自动执行            用户确认触发              AI整理完成后
```

---

## 阶段1: 初步整理（5:00 自动）

**目标**: 确保完整性，不遗漏任何对话

**执行脚本**: `process_raw.py`

**行为**:
- ✅ 扫描近3天所有会话文件
- ✅ 提取完整对话内容
- ✅ **本地保存**到 `obsidian-vault/02-Conversations/YYYY-MM-DD_conversations.md`
- ❌ **不推送 GitHub**

**输出示例**:
```
obsidian-vault/02-Conversations/
├── 2026-03-19_conversations.md      # 原始完整对话（本地）
└── 2026-03-18_conversations.md      # 原始完整对话（本地）
```

---

## 阶段2: AI 深度整理（用户确认后）

**目标**: 主题精炼、信息密度高、有价值

**触发方式**:
- 用户回复 "整理"
- 用户指定 "处理昨天对话"

**执行流程**:
1. **读取**阶段1生成的 `*_conversations.md`
2. **AI 分析**: 主题识别 → 核心提炼 → 价值提取
3. **生成** `*_主题整理版.md`
4. ❌ **不推送 GitHub**（等待阶段3统一推送）

**输出示例**:
```
obsidian-vault/02-Conversations/
├── 2026-03-19_conversations.md          # 原始（本地）
├── 2026-03-19_主题整理版.md              # AI精炼（本地）
└── 2026-03-18_conversations.md          # 原始（本地）
```

### AI 整理质量标准

| 维度 | 要求 |
|-----|------|
| **主题完整性** | 不遗漏重要讨论话题 |
| **信息密度** | 去除重复、空洞、操作细节 |
| **价值提炼** | Key Takeaway + 核心观点 + 思考延伸 |
| **结构化** | 主题分类清晰，层次分明 |

---

## 阶段3: 推送到 GitHub（AI整理完成后）

**目标**: 仓库只保留最终有价值的内容

**触发时机**:
- 所有AI整理任务完成后
- 定时任务（如每日 8:30 复盘前）

**推送内容**:
- ✅ `*_主题整理版.md`（必须）
- ⚠️ `*_conversations.md`（可选，看用户偏好）
- ❌ 中间迭代版本（绝不推送）

**Git 提交规范**:
```bash
git add obsidian-vault/02-Conversations/*_主题整理版.md
git commit -m "content: YYYY-MM-DD 对话主题整理

- 主题数: X 个
- 核心洞察: XXX
- 价值点: XXX"
git push origin main
```

## 工作流程对比

### 旧流程（问题）
```
5:00 初步整理 → 推送GitHub
    ↓
8:30 发现遗漏 → 再次推送GitHub
    ↓
用户查看 → 发现格式问题 → 又推送GitHub
```
**问题**: GitHub 仓库有3次+提交，历史混乱

### 新流程（优化）
```
5:00 初步整理 → 本地保存（不推送）
    ↓
用户确认 → AI深度整理 → 本地生成（不推送）
    ↓
整理完成 → 一次性推送GitHub（最终版）
```
**优势**: GitHub 只有1次提交，历史清晰

## 执行检查清单

### 阶段1执行后检查
- [ ] 文件已保存到本地 `02-Conversations/`
- [ ] 消息数量合理（非0，非异常少）
- [ ] **未推送** GitHub

### 阶段2执行后检查
- [ ] 主题识别完整
- [ ] 信息密度高（去除碎片）
- [ ] 有价值提炼（Key Takeaway明确）
- [ ] **未推送** GitHub

### 阶段3执行后检查
- [ ] 只推送最终版本
- [ ] 提交信息清晰
- [ ] 推送成功验证

---

## 异常处理

| 场景 | 处理方案 |
|-----|---------|
| 阶段1发现消息过少 | 标记"待补偿"，8:30复盘前重试 |
| 阶段2AI整理超时 | 拆分主题分批处理，完成后统一推送 |
| 阶段3推送失败 | 记录失败原因，下次整理时重试 |
